## 1. `platformdb` вҖ” Onboarding & Metadata

Owned by `platform-api`. This database manages user identities, organizations, invite status, and audit logs.

### Schema Management
* **Technology:** **Drizzle ORM**
* **Migration Run:** Driven by the one-shot container `platform-migrate` on startup using `drizzle-kit push`.

### Key Tables
* **`users`** вҖ” Accounts, passwords (bcrypt-hashed), email address.
* **`organizations`** вҖ” Multi-tenant organization boundaries.
* **`memberships`** вҖ” Links users to organizations and assigns roles (`owner`, `admin`, `member`).
* **`anchors`** вҖ” Anchor metadata stubs (name, domain slug, status: draft -> active).
* **`provisioning_jobs`** вҖ” Track progress stage (`stage` status: e.g. "Generating keys", "completed") and any failures.

---

## 2. `controldb` вҖ” Provisioner Registry

Owned by `anchor-service/control-plane`. It tracks deployed containers and handles key vault security.

### Schema Management
* **Technology:** Managed directly by the control plane using simple schema creations and Express db client connections.

### Key Tables
* **`tenants`** вҖ” Maps slugs to container IDs (`ap_container_id`, `biz_container_id`), domains, and status.
* **`anchor_secrets`** вҖ” Encrypted seeds (`S...`) and plaintext public keys (`G...`) for Issuer, Distributor, and Signing accounts.

---

## 3. `aggregatordb` вҖ” Discovery Registry

Owned by the `aggregator-service`.

### Key Tables
* **`anchors`** вҖ” Registered anchors, fee percentages, limits, regions, and current availability status.
* **`quotes`** вҖ” Active transaction quotes with TTL values.
* **`routing_decisions`** вҖ” Logs which routes were selected and why.

---

## 4. `anchordb_<slug>` вҖ” Anchor Ledger

Every active anchor gets its own isolated database. It holds two schemas:
1. **`public` (Anchor Platform):** Holds Flyway-managed protocol tables for transactions and ledger offsets.
2. **`nordstern` (Business Server):** Handles money-safety outboxes and audit details:
   * **`deposit_releases`:** Lock outbox tracking Transfer-After-Commit deposit statuses.
   * **`withdrawal_payouts`:** Lock table ensuring At-Most-Once payouts.
   * **`kyc_verifications`:** Anchor-local KYC approval logs.
